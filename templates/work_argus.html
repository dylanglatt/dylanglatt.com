The Problem

There are public databases that log hundreds of thousands of global events every day — protests, military actions, political violence, everything. The data is free and open. But the vast majority is noise: duplicate reports, bad extractions, non-conflict events mislabeled as conflict. Platforms like Palantir solve this at scale for governments and institutions. I wanted to see how far I could get building the same kind of thing from scratch, using only public data and API integrations.

What I Built

A real-time conflict event monitor. Raw event data comes in every 15 minutes, gets run through a five-stage filtering pipeline, and what comes out is a clean feed of verified events on a live map. You can see what’s happening, where, how severe it is, and whether satellite data backs it up.

How It Works

The first stage is rule-based: filter by event type, check if known military or rebel groups are involved, require multiple independent sources, and block unreliable domains. That alone eliminates roughly 80% of the noise before anything expensive runs.

Events that are clearly significant pass through automatically. Ambiguous ones go to an AI classifier that reads the underlying news article and scores the event on five dimensions: credibility, severity, specificity, novelty, and conflict relevance. This keeps AI costs low — it only evaluates the cases that rules can’t confidently decide.

Events that survive are then cross-referenced against NASA satellite thermal imagery. If a reported explosion or battle lines up with a heat signature at that location, it gets flagged as independently confirmed — verification that doesn’t depend on media reporting at all.

Before going live, I validated the pipeline through data analysis on two years of historical conflict records — making sure the system was catching real events, not just surfacing noise with high confidence.

The Result

Live at argusosint.vercel.app. Built from scratch: I designed the data pipeline, wrote the Python automation for ingestion and filtering, integrated the satellite layer, built the frontend, and handled deployment — all from public data, on my own.

Data Sources

GDELT 2.0 Public event database, updated every 15 minutes — logs hundreds of thousands of events per day across 100+ countries from global news sources
POLECAT Government-funded historical conflict dataset — two years of validated records used to backtest the filtering pipeline before launch
NASA FIRMS Satellite thermal imagery, updated near real-time — matched to reported battles and explosions by location for independent confirmation
UN ReliefWeb Humanitarian situation reports by country — pulled live and surfaced in the country brief panel
CAMEO / Goldstein Standard classification systems for event types and severity — used to categorize and rank events before AI filtering

Stack

React 19 + Mapbox GL JS Frontend analyst workstation — clustered event map, sortable feed, 6-hour time-series chart, and escalation detection UI
Express.js API layer serving filtered events, satellite data, and user feedback
Claude Haiku (Anthropic API) AI noise filter — reads news articles for ambiguous events and decides whether they represent real conflict or should be discarded
Python data pipeline Automated event ingestion, normalization, satellite matching, and historical data analysis
Vercel + Vercel Blob Deployment and analyst feedback persistence
